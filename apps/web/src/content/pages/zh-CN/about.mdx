## 技术栈和偏好

以下是我在项目中经常使用的一些工具和选择：- **操作系统**

- 服务器上使用 **Ubuntu Server 24.04 LTS**
- 主力笔记本电脑使用 **Windows 10 Pro**
- 实验用的 Arch 衍生发行版，如 **EndeavourOS**、**CachyOS** 和 **Garuda**

- **网络和访问**
  - 使用 **Tailscale** 进行安全的远程访问
  - 使用 **Cloudflared** 发布服务，无需配置路由器的端口转发

- **容器和服务**
  - 使用 **Docker** 和 **Docker Compose**（插件，而非旧的 `docker-compose` 命令）
  - `docker-compose.yml` 文件不再包含旧的 `version` 字段
  - 使用外部 **PostgreSQL**，而非每个容器一个数据库
  - 容器中使用固定 DNS：
    - `1.1.1.1`
    - `8.8.8.8`

- **编程和工具**
  - 使用 **Python** 进行自动化和脚本编写
  - 使用 `pipx` 全局但干净地安装基于 Python 的工具
  - 使用 Shell 脚本处理简单任务

- **数据库**
  - 实验室中使用 **PostgreSQL** 作为主要数据库
  - 对 **MongoDB** 和其他数据库引擎感兴趣，以比较不同的方法

- **AI 和自动化**
  - 在 Ubuntu 上直接在主机（而非容器）中运行 **Ollama**
  - 测试不同的模型和提示
  - 计划开发个人自动化和助手

我喜欢可复现的设置。如果我编写一个 Compose 文件或脚本，我会尽量确保几个月后能够在不猜测的情况下重建相同的环境。

## 实验室、仓库和文档风格

我的一个主要长期项目是构建一个 **DBA 实验室和作品集**，专注于 **PostgreSQL**。

在我的仓库中，你可以期待以下内容：

- 每个服务或实验室的结构化目录，例如：
  - `~/docker/docker-compose/{service}`
  - `~/docker/docker-data/{service}`
- `docker compose` 文件：
  - 避免使用弃用的选项
  - 在合理的情况下使用外部数据库
  - 包含示例环境变量值，以明确各项配置的含义
- 使用 Markdown 编写的文档，通常以第一人称撰写，包含以下部分：
  - 变更了什么
  - 如何变更的
  - 为什么变更
- 代码中的注释使用**小写**
- SQL 查询主要使用**小写**
- 偏好开源许可证，公开项目中经常使用 **LGPL 2.1**

我喜欢将这些实验室视为迷你生产环境。即使没有其他人使用它们，我也会同时扮演 DBA、系统管理员和值班人员的角色。
