## ホームラボとインフラストラクチャ

私のホームラボは私の最高の先生だと言うのが好きです。

そこにあるものの一部：

- **Ubuntu Server 24.04 LTS**を実行しているホームサーバー、以下を含む：
  - データベースとサービスに特化した高性能マシン
  - 実験用にまだ使用している軽量マシン
- エクストラノードやデータベースホストとして機能する**Oracle Cloud**インスタンス

ここで登場したり消えたりする典型的なサービス：

- ファイル用のNextcloud
- メモ用のJoplinサーバー
- クイックテキスト共有用のPrivateBin
- セルフホスト型モニタリングツール
- Ollamaを通じたローカルAIモデル
- ワークフローをサポートする小規模なWebアプリ

私はドメイン**`yuricunha.com`**を所有しており、多くのサービスが`something.yuricunha.com`のようなサブドメインで動作しています。このサイトはそのすべての見える顔の一つです。

## テックスタックと選好

これらは私のプロジェクトでよく登場するツールや選択肢です：- **オペレーティングシステム**
  - サーバー上の **Ubuntu Server 24.04 LTS**
  - メインラップトップの **Windows 10 Pro**
  - 実験用に Arch ベースのディストリビューション（EndeavourOS、CachyOS、Garuda など）

- **ネットワーキングとアクセス**
  - セキュアなリモートアクセスに **Tailscale**
  - ルーターのポートフォワーディングを触らずにサービスを公開するための **Cloudflared**

- **コンテナとサービス**
  - **Docker with Docker Compose**（古い `docker-compose` コマンドではなくプラグイン）
  - 古い `version` キーのない `docker-compose.yml` ファイル
  - コンテナごとのデータベースではなく、外部の **PostgreSQL**
  - コンテナ内の固定 DNS に以下を使用：
    - `1.1.1.1`
    - `8.8.8.8`

- **プログラミングとツール**
  - 自動化とスクリプトに **Python**
  - Python ベースのツールをグローバルかつクリーンにインストールするための `pipx`
  - 簡単なタスクに **シェルスクリプト**

- **データベース**
  - ラボ環境でのメインデータベースとして **PostgreSQL**
  - アプローチの比較のために **MongoDB** やその他のエンジンにも興味あり

- **AI と自動化**
  - コンテナではなく、Ubuntu ホスト上で直接実行される **Ollama**
  - さまざまなモデルとプロンプトのテスト
  - 個人的な自動化とアシスタントの計画

私は再現可能なセットアップを好みます。`compose` ファイルやスクリプトを書く際は、数か月後に推測せずに同じ環境を再現できるようにしています。

## ラボ、リポジトリ、ドキュメントスタイル

私の主要な長期プロジェクトの1つは、**DBA ラボとポートフォリオ**の構築で、特に **PostgreSQL** に焦点を当てています。

私のリポジトリから期待できること：

- 各サービスやラボごとに構造化されたディレクトリ、例えば：
  - `~/docker/docker-compose/{service}`
  - `~/docker/docker-data/{service}`
- `docker compose` ファイルは：
  - 非推奨のオプションを避ける
  - 外部データベースを使用する（適切な場合）
  - 環境変数のサンプル値を含め、どこに何が入るか明確にする
- マークダウンで書かれたドキュメントで、通常は一人称で、以下のセクションを含む：
  - 何が変わったか
  - どのように変わったか
  - なぜ変わったか
- コード内のコメントは**小文字**
- SQL クエリは主に**小文字**
- オープンソースライセンスを好み、**LGPL 2.1** を公開プロジェクトでよく使用

これらのラボをミニプロダクション環境として扱うことを好みます。他の誰も使わなくても、自分が DBA、シス管理者、オンコール担当者であるかのように振る舞います。
