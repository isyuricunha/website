## meu homelab e infraestrutura

eu gosto de dizer que meu homelab é meu melhor professor.

algumas coisas que vivem lá:

* servidores em casa rodando **ubuntu server 24.04 lts**, incluindo:

  * uma máquina mais potente focada em bancos de dados e serviços
  * uma máquina mais leve que eu ainda uso para experimentos
* instâncias na **oracle cloud** que atuam como nós extras e hosts de banco de dados

* túneis **cloudflared** para expor serviços sem abrir portas aleatórias na internet
* **portas externas aleatórias** mapeadas para containers, em vez das padrões, como uma camada extra pequena contra bots
* um servidor dedicado de banco na oracle onde eu centralizo bancos como postgresql e outros

serviços típicos que aparecem e somem por aqui:

* nextcloud para arquivos
* joplin server para notas
* privatebin para compartilhar texto rapidamente
* ferramentas de monitoramento self-hosted
* modelos de ia local via ollama
* pequenos web apps para apoiar meu workflow

eu tenho o domínio **`yuricunha.com`**, e muitos serviços vivem em subdomínios como `something.yuricunha.com`. este site é uma das faces visíveis de tudo isso.

## stack e preferências

estas são algumas ferramentas e escolhas que aparecem bastante nos meus projetos:

* **sistemas operacionais**

  * ubuntu server 24.04 lts nos servidores
  * windows 10 pro no meu notebook principal
  * distros baseadas em arch para experimentos, como endeavouros, cachyos e garuda
* **rede e acesso**

  * tailscale para acesso remoto seguro
  * cloudflared para publicar serviços sem mexer com port forwarding no roteador
* **containers e serviços**

  * docker com **docker compose** (o plugin, não o comando antigo `docker-compose`)
  * arquivos `docker-compose.yml` sem a chave antiga `version`
  * postgresql externo em vez de um banco por container
  * dns fixo nos containers usando:

    * `1.1.1.1`
    * `8.8.8.8`
* **programação e ferramentas**

  * python para automação e scripts
  * `pipx` para instalar ferramentas python globalmente, mas de forma limpa
  * shell scripts para tarefas simples
* **bancos de dados**

  * postgresql como o principal banco nos meus labs
  * interesse em mongodb e outros engines para comparar abordagens
* **ia e automação**

  * ollama rodando no ubuntu, direto no host, não em containers
  * testes com diferentes modelos e prompts
  * planos para automações e assistentes pessoais

eu gosto de setups que sejam reproduzíveis. se eu escrevo um compose file ou um script, eu tento fazer de um jeito que eu consiga recriar o mesmo ambiente meses depois, sem chute.

## labs, repositórios e estilo de documentação

um dos meus principais projetos de longo prazo é construir um **lab e portfólio de dba**, focado em postgresql.

o que você pode esperar dos meus repositórios:

* diretórios estruturados para cada serviço ou lab, por exemplo:

  * `~/docker/docker-compose/{service}`
  * `~/docker/docker-data/{service}`
* arquivos de `docker compose` que:

  * evitam opções depreciadas
  * usam bancos externos quando faz sentido
  * incluem valores de exemplo para variáveis de ambiente, para ficar claro o que vai onde
* documentação em markdown, geralmente em primeira pessoa, com seções explicando:

  * o que mudou
  * como mudou
  * por que mudou
* comentários no código em **minúsculas**
* queries sql majoritariamente em **minúsculas**
* preferência por licenças open source, muitas vezes **lgpl 2.1** em projetos públicos

eu gosto de tratar esses labs como mini ambientes de produção. mesmo que ninguém mais use, eu ajo como se eu fosse o dba, o sysadmin e a pessoa de plantão ao mesmo tempo.
