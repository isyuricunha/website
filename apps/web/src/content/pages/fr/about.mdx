## mon homelab et mon infrastructure

J'aime dire que mon homelab est mon meilleur professeur.

Voici quelques éléments qui y vivent :

- Des serveurs domestiques exécutant **Ubuntu Server 24.04 LTS**, notamment :
  - Une machine plus puissante axée sur les bases de données et les services.
  - Une machine plus légère que j'utilise encore pour des expériences.

Voici quelques services typiques qui apparaissent et disparaissent ici :

- Nextcloud pour les fichiers.
- Joplin Server pour les notes.
- PrivateBin pour le partage rapide de texte.
- Des outils de surveillance auto-hébergés.
- Des modèles d'IA locaux via Ollama.

## pile technologique et préférences

Voici quelques outils et choix qui apparaissent souvent dans mes projets :- **systèmes d'exploitation**

- ubuntu server 24.04 lts sur les serveurs
- windows 10 pro sur mon ordinateur portable principal
- distributions basées sur arch pour des expériences, comme endeavouros, cachyos et garuda

- **réseau et accès**
  - tailscale pour un accès distant sécurisé
  - cloudflared pour publier des services sans toucher à la redirection de ports du routeur

- **conteneurs et services**
  - docker avec **docker compose** (le plugin, pas l'ancienne commande `docker-compose`)
  - fichiers `docker-compose.yml` sans l'ancienne clé `version`
  - base de données postgresql externe au lieu d'une base de données par conteneur
  - dns fixe dans les conteneurs en utilisant :
    - `1.1.1.1`
    - `8.8.8.8`

- **programmation et outils**
  - python pour l'automatisation et les scripts
  - `pipx` pour installer globalement mais proprement des outils basés sur python
  - scripts shell pour des tâches simples

- **bases de données**
  - postgresql comme base de données principale dans mes labs
  - intérêt pour mongodb et d'autres moteurs pour comparer les approches

- **ia et automatisation**
  - ollama exécuté sur ubuntu, directement sur l'hôte, pas dans des conteneurs
  - tests avec différents modèles et prompts
  - projets d'automatisations personnelles et d'assistants

J'aime les configurations reproductibles. Si j'écris un fichier compose ou un script, j'essaie de le faire de manière à pouvoir recréer le même environnement des mois plus tard sans devoir deviner.

## labs, dépôts et style de documentation

L'un de mes principaux projets à long terme est la création d'un **lab et portfolio dba**, axé sur postgresql.

Voici ce que vous pouvez attendre de mes dépôts :

- des répertoires structurés pour chaque service ou lab, par exemple :
  - `~/docker/docker-compose/{service}`
  - `~/docker/docker-data/{service}`
- des fichiers `docker compose` qui :
  - évitent les options obsolètes
  - utilisent des bases de données externes lorsque cela a du sens
  - incluent des valeurs d'environnement exemples, pour clarifier ce qui va où
- une documentation écrite en markdown, généralement à la première personne, avec des sections expliquant :
  - ce qui a changé
  - comment cela a changé
  - pourquoi cela a changé
- des commentaires dans le code en **minuscules**
- des requêtes sql principalement en **minuscules**
- une préférence pour les licences open source, souvent **lgpl 2.1** dans les projets publics

J'aime traiter ces labs comme des environnements de production miniatures. Même si personne d'autre ne les utilise, j'agis comme si j'étais à la fois le dba, le sysadmin et la personne d'astreinte.

## style d'étude et état d'esprit

Comment j'aime étudier et progresser dans ce domaine :

- mélanger **théorie et pratique** en permanence
- casser des choses exprès, puis les réparer
- garder des **notes détaillées** des commandes, des sorties et des décisions
- créer des **runbooks** pour les tâches répétitives ou stressantes, par exemple :
  - restaurer des sauvegardes
  - reconfigurer un cluster
  - effectuer de la maintenance avec un temps d'arrêt minimal
- se concentrer sur **l'observabilité** :
  - logs
  - métriques
  - alertes
- revisiter d'anciens labs avec de nouvelles connaissances pour voir ce que je ferais différemment maintenant

Je préfère comprendre comment les choses fonctionnent sous la surface plutôt que mémoriser une liste de commandes. Si je sais où se trouvent les logs, comment le processus démarre, comment les données sont stockées et ce que fait la configuration, je me sens beaucoup plus à l'aise pour toucher aux systèmes de production.
