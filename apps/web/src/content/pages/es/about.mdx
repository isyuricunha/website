## mi homelab e infraestructura

Me gusta decir que mi homelab es mi mejor maestro.

Algunas cosas que viven allí:

- Servidores en casa ejecutando **Ubuntu Server 24.04 LTS**, incluyendo:
  - Una máquina más potente enfocada en bases de datos y servicios.
  - Una máquina más ligera que aún uso para experimentos.

- Un servidor de bases de datos dedicado en Oracle donde centralizo bases de datos como PostgreSQL y otras.

Servicios típicos que aparecen y desaparecen aquí:

- Nextcloud para archivos.
- Servidor Joplin para notas.
- PrivateBin para compartir texto rápidamente.
- Herramientas de monitoreo autoalojadas.
- Modelos de IA locales a través de Ollama.
- Pequeñas aplicaciones web para apoyar mi flujo de trabajo.

Soy dueño del dominio **`yuricunha.com`**, y muchos servicios viven en subdominios como `algo.yuricunha.com`. Este sitio es una de las caras visibles de todo eso.

## stack tecnológico y preferencias

Estas son algunas herramientas y elecciones que aparecen frecuentemente en mis proyectos:- **sistemas operativos**

- ubuntu server 24.04 lts en servidores
- windows 10 pro en mi laptop principal
- distros basadas en arch para experimentos, como endeavouros, cachyos y garuda
- **redes y acceso**
  - tailscale para acceso remoto seguro
  - cloudflared para publicar servicios sin tocar el reenvío de puertos del router
- **contenedores y servicios**
  - docker con **docker compose** (el plugin, no el antiguo comando `docker-compose`)
  - archivos `docker-compose.yml` sin la antigua clave `version`
  - postgresql externo en lugar de una base de datos por contenedor
  - dns fijo en contenedores usando:
    - `1.1.1.1`
    - `8.8.8.8`
- **programación y herramientas**
  - python para automatización y scripts
  - `pipx` para instalar herramientas basadas en python de manera global pero limpia
  - scripts de shell para tareas simples
- **bases de datos**
  - postgresql como base de datos principal en mis laboratorios
  - interés en mongodb y otros motores para comparar enfoques
- **ia y automatización**
  - ollama ejecutándose en ubuntu, directamente en el host, no en contenedores
  - pruebas con diferentes modelos y prompts
  - planes para automatizaciones y asistentes personales

me gustan las configuraciones que son reproducibles. si escribo un archivo compose o un script, intento hacerlo de manera que pueda recrear el mismo entorno meses después sin adivinar.

## laboratorios, repositorios y estilo de documentación

uno de mis proyectos principales a largo plazo es construir un **laboratorio y portafolio de dba**, enfocado en postgresql.

cosas que puedes esperar de mis repositorios:

- directorios estructurados para cada servicio o laboratorio, por ejemplo:
  - `~/docker/docker-compose/{servicio}`
  - `~/docker/docker-data/{servicio}`
- archivos `docker compose` que:
  - evitan opciones obsoletas
  - usan bases de datos externas cuando tiene sentido
  - incluyen valores de entorno de ejemplo, para que quede claro qué va dónde
- documentación escrita en markdown, generalmente en primera persona, con secciones que explican:
  - qué cambió
  - cómo cambió
  - por qué cambió
- comentarios en el código en **minúsculas**
- consultas sql mayormente en **minúsculas**
- preferencia por licencias de código abierto, a menudo **lgpl 2.1** en proyectos públicos

me gusta tratar estos laboratorios como mini entornos de producción. incluso si nadie más los usa, actúo como si fuera el dba, el sysadmin y la persona de guardia al mismo tiempo.
